Why priority matters

Not every issue has the same impact.

Some problems block access. Others are important but less urgent. The order matters because time and budget are limited.

Blockers first

If a user cannot log in or submit work, the task fails immediately.

Frequent paths next

Search, filters, and overview screens are used repeatedly and affect many users.

Supportive items last

Progress, messages, and downloads still matter, but they rarely block access by themselves.

All issues still matter. Priority only decides the order of the work.
Explain that prioritization is a risk-reduction tool. It does not excuse remaining issues; it simply organizes the fix sequence.

Decision logic

Use risk, frequency, and reach to rank components.

A component becomes more urgent when it blocks the core flow, appears often, or affects many screens at once.

P1 Blocks access or the core task

Fix first when the user cannot proceed without it.

P2 Used often or drives navigation

Improve early when the component appears in many daily paths.

P3 Supports status or secondary tasks

Still important, but usually after the main flow is stable.

Structure

Is the component organized clearly?

Labels

Does the wording explain the action?

Keyboard

Can the component be used without a mouse?

Contrast

Can users read and recognize it easily?

Use these rules to rank the examples. The aim is not to guess a perfect answer, but to defend a sensible order.

Task assignment

Task Assignment

Use the priority logic from the previous slides.

Task 2: Prioritize accessible components
Participants receive a scenario for a digital course platform. The platform contains Login, Course overview, Search, Filters, Progress indicator, Assignment form, Instructor messages, and a Download area.

Assignment:
1. List the most important web components on the platform.
2. Evaluate which components are especially barrier-sensitive.
3. Check each component using structure, labels, keyboard operation, contrast, error messages, and understandability.
4. Prioritize the components according to risk and frequency of use.
5. Decide which three components should be improved first.
6. Justify the prioritization from user, quality, and risk perspectives.
7. Name the information still missing for a safe evaluation.

Objective:
Participants practice treating accessibility as a prioritizable quality and risk area. They learn to make meaningful improvement decisions under limited time and budget.

Expected output:
A prioritized component list, short reasoning, and missing information.
